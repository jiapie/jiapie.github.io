# 02.生成式 AI 能力強化

這部分考企業內部如何讓 AI 變聰明，並處理髒資料。

- 資料預處理(Data Preprocessing)：
    - **資料清理**：去除重複段落、修正語法錯誤、英文詞幹還原。核心目標是 **「去除雜訊、降低維度、統一格式」**
        - 標記並修正語法錯誤。
        - 去除重複的句子或段落。
        - 將文字轉換為詞幹形式。
    - **資料標註**：額外幫資料打標籤，例如增加文字情感標註。這屬於特徵工程，不屬於資料清理。
    - **資料集成**：將多源資料整合成統一視角，用於建置資料湖泊或資料倉儲。
        - **注意：這無法幫企業「定義」痛點。**

- 企業導入兩大主流戰略(LLM 客製化)：
    - RAG(檢索增強生成)：外掛知識庫。不改變模型參數，直接去私有資料庫(如資料湖泊)抓資料給 AI 看再回答。成本低、即時性高。
    - Fine-tuning(微調)：改變模型內部的權重參數，讓 AI 徹底學會某種特定的語氣、格式或專業領域知識。成本高。

---
## RAG 詳細架構

**三階段流程：**

```
使用者查詢 → [檢索階段] 從知識庫搜尋相關文件
           → [整合階段] 將檢索結果與查詢合併為完整上下文
           → [生成階段] LLM 根據上下文生成回答（避免幻覺）
```

**RAG 的核心優勢：** 知識庫即時更新，模型無需重新訓練。企業私有文件（PDF、內部 Wiki）可即插即用。

---

## Fine-tuning 詳細

**核心概念：** 在既有預訓練模型基礎上，用特定領域資料再訓練，更新模型參數。

**參數高效微調（PEFT）技術：**

| 方法 | 說明 | 優勢 |
|------|------|------|
| **LoRA**（低秩適應矩陣） | 只在少數矩陣插入低秩分解，僅更新少量參數 | 大幅降低顯存與計算成本 |
| **Prefix Tuning** | 在輸入前加入可學習的前綴向量 | 原始模型參數不變 |
| **Adapter** | 在各層插入小型可訓練模組 | 模組化、易替換 |

**Fine-tuning 風險：**
- **過擬合**：資料量過少時只記住範例，泛化能力下降
- **遺災性遺忘（Catastrophic Forgetting）**：在新任務表現提升，但在舊能力上退化
- **解決方案**：搭配正則化、驗證集評估、持續學習機制

---

## RAG vs Fine-tuning 選擇指南 **[必考]**

| | **RAG** | **Fine-tuning** |
|--|---------|----------------|
| **知識更新** | 即時（更新知識庫即可）| 需重新訓練 |
| **成本** | 低 | 高（需標注資料＋算力）|
| **適合場景** | 知識更新快（法律、金融、客服）| 固定專業領域（醫療診斷、術語翻譯）|
| **幻覺風險** | 低（有文件來源依據）| 較高（依賴訓練資料記憶）|
| **客製程度** | 中（輸出風格由原模型決定）| 高（語氣、格式可完全客製）|

> **最佳實務：** 大多數產業應用中，**RAG + Fine-tuning 混合使用**，兼顧即時性與專業性。

## AI Agent 智慧代理

> AI Agent 是生成式 AI 能力強化的最高形態——從「問答工具」進化為「自主執行者」。

**六大核心能力：**

| 能力 | 說明 |
|------|------|
| **感知（Perception）** | 接收環境資訊與任務輸入 |
| **推理（Reasoning）** | 分析資訊並規劃解決方案 |
| **行動（Action）** | 執行決策並產生結果 |
| **工具使用（Tool Use）** | 調用外部工具與 API（搜尋、計算、資料庫查詢）|
| **記憶（Memory）** | 儲存歷史資訊以優化未來決策 |
| **協作（A2A）** | 代理間互相協作，形成多智能體系統 |

**技術基礎：** LLM + 工具使用 + 記憶模組 + 強化學習

**應用：** 智慧客服、銷售助理、文件自動處理、智慧工廠流程最佳化

## RAG 進階技術

### Chunking（文件分塊）

**問題：** 直接以整份長篇文件進行向量檢索，模型回覆常包含無關內容，引用段落不精準。

**解法：** 將文件切分為較小的語意單元（Chunk）再建立向量索引。

| 分塊策略 | 說明 | 適用場景 |
|---------|------|---------|
| **固定長度分塊** | 按固定字元/Token 數切割 | 簡單快速，通用場景 |
| **語意分塊** | 依段落、句子邊界切割 | 保留語意完整性 |
| **滑動視窗** | 相鄰塊有重疊（Overlap）| 避免資訊在邊界被截斷 |

**Chunking 主要目的：** 提升檢索結果的語意對齊程度，降低長文件帶來的干擾。

---

### 增量索引更新（Incremental Indexing）

**問題：** 傳統 RAG 每次知識庫更新需重建完整索引，導致服務中斷。

**解法：** 導入支援**增量更新**的索引管理方式，使異動文件可即時反映於檢索結果，不影響其他文件的現有索引。

---

### MCP（Model Context Protocol）

**定義：** 由 Anthropic 提出的開放標準協議，讓 AI 模型（LLM）能夠標準化地與**外部工具、系統、資料源**互動。

**MCP vs RAG 的差異：**

| | **MCP** | **RAG** |
|--|---------|---------|
| **定位** | 標準化通訊協議，規範模型與工具的互動方式 | 補充模型外部知識的技術架構 |
| **著重** | 模型與**外部工具/系統**的互動（如 API、資料庫、程式執行）| 從知識庫**檢索相關文件**輔助生成 |
| **類比** | USB-C 標準（統一介面）| 圖書館查書系統 |

> **考試陷阱：** MCP 是通訊**協議**（Protocol），不是資料檢索技術；RAG 是**知識補充**技術，不是模型與工具互動的協議。

## LLM 架構與成本

### Encoder-Decoder vs Decoder-only

| | **Encoder-Decoder** | **Decoder-only** |
|--|---------------------|-----------------|
| **結構** | 編碼器理解輸入 + 解碼器生成輸出 | 單一模型同時處理上下文與生成 |
| **機制** | 區分「輸入理解」與「內容生成」兩階段 | 以自迴歸方式逐 Token 生成 |
| **代表模型** | BERT（Encoder only）、T5、BART | GPT 系列、LLaMA、Claude |
| **適合任務** | 機器翻譯、摘要（需充分理解輸入）| 對話、文本生成、程式碼生成 |

---

### Token Economics（Token 經濟學）

**Token：** LLM 處理的最小語言單元（約等於半個英文單字或一個中文字）

**推論成本組成：**

| 項目 | 說明 |
|------|------|
| **輸入 Token（Prompt）** | 每次 API 呼叫所需的輸入 Token 數量 × 單價 |
| **輸出 Token（Completion）** | 生成回答所消耗的 Token × 單價（通常比輸入貴）|
| **推論 GPU 成本** | 模型推理過程的 GPU 使用費 |

> **不屬於** Token Economics 的範疇：模型**訓練**階段使用的 GPU 記憶體成本（那是預訓練成本）

**ROI 計算範例：**
- 原成本：600 則 × 50 元/則 = 30,000 元/月
- AI 成本：600 則 × 2,000 Token × 0.8元/1,000 Token = 960 元/月
- 每月節省：30,000 − 960 = **29,040 元**
- 整合成本 200,000 ÷ 29,040 ≈ **7 個月回收**

### 隱私增強技術

| 技術 | 原理 | 適用場景 |
|------|------|---------|
| **同態加密（Homomorphic Encryption）** | 在**加密狀態下**仍可進行計算，計算結果解密後與明文計算相同 | 需要在不洩漏原始資料的前提下完成模型計算 |
| **聯邦學習（Federated Learning）** | 資料不離開本地，各節點訓練後只上傳模型梯度 | 資料分散且不可集中（醫療跨院訓練）|
| **差分隱私（Differential Privacy）** | 加入數學雜訊，使查詢結果無法還原個別記錄 | 統計資料發布、大規模資料收集 |
| **零知識證明（ZKP）** | 在不揭露資訊內容的前提下，證明某命題為真 | 身份驗證、合規性證明 |

> **考試陷阱：** 「同態加密」= 資料加密狀態下仍可計算，適合「原始資料不得外洩，但仍需完成模型計算」的場景。
