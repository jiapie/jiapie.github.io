# 02. 資訊安全防護實務

> iPAS 資訊安全工程師【中級】考科二  
> 考試時間：90 分鐘 / 單複選題 + 情境題組，共 40 題

## 評鑑主題一：弱點、攻擊與防護實務

### 1-1 弱點、威脅分類與攻擊手法

#### 弱點分類與評估

**CVE / NVD**
- **CVE（Common Vulnerabilities and Exposures）**：唯一識別已知弱點的標準編號，格式：`CVE-年份-編號`
- **NVD（National Vulnerability Database）**：美國 NIST 維護，提供 CVE 詳細資訊與 CVSS 分數

**CWE（Common Weakness Enumeration）**
- 軟體弱點分類系統，描述根本弱點類型（非特定 CVE）
- 常見 CWE：
  - CWE-79：跨站腳本（XSS）
  - CWE-89：SQL Injection
  - CWE-20：輸入驗證不足
  - CWE-798：硬式編碼憑證

**OWASP Top 10（2021）**
| 排名 | 弱點類型 |
|---|---|
| A01 | 存取控制失效（Broken Access Control） |
| A02 | 密碼學失效（Cryptographic Failures） |
| A03 | 注入攻擊（Injection） |
| A04 | 不安全設計（Insecure Design） |
| A05 | 安全設定錯誤（Security Misconfiguration） |
| A06 | 過時或不安全的元件（Vulnerable and Outdated Components） |
| A07 | 身份識別與驗證失效（Identification and Authentication Failures） |
| A08 | 軟體與資料完整性失效（Software and Data Integrity Failures） |
| A09 | 安全記錄與監控失效（Security Logging and Monitoring Failures） |
| A10 | 伺服器端請求偽造（SSRF） |

---

#### 網路層攻擊手法

**DoS / DDoS**
- **DoS（阻斷服務）**：單一來源大量請求癱瘓服務
- **DDoS（分散式阻斷服務）**：透過殭屍網路（Botnet）多來源攻擊
- 類型：
  - 流量型（Volumetric）：UDP Flood、ICMP Flood
  - 協定型（Protocol）：SYN Flood、Ping of Death
  - 應用層型：HTTP Flood、Slowloris

**中間人攻擊（MITM, Man-in-the-Middle）**
- ARP Spoofing：偽造 ARP 回應，攔截同網段流量
- SSL Stripping：將 HTTPS 降級為 HTTP
- 防禦：HTTPS + HSTS、憑證固定（Certificate Pinning）、Dynamic ARP Inspection（DAI）

**DNS 攻擊**
- DNS Spoofing / Cache Poisoning：污染 DNS 快取，導向惡意網站
- DNS Amplification：利用 DNS 放大反射流量（DDoS）
- 防禦：DNSSEC、DNS over HTTPS（DoH）

**封包嗅探（Sniffing）**
- 被動監聽網路流量（Wireshark）
- 防禦：加密所有傳輸（TLS）、使用交換器替代集線器

#### 應用層攻擊手法

**SQL Injection（SQLi）**
- 原理：將惡意 SQL 語法插入輸入欄位，操控資料庫
- 類型：Classic、Blind（Boolean/Time-based）、Error-based、Out-of-band
- 防禦：Prepared Statement（參數化查詢）、WAF、最小權限資料庫帳號

**跨站腳本（XSS, Cross-Site Scripting）**
- Reflected XSS：惡意腳本透過 URL 反射給受害者
- Stored XSS：惡意腳本持久儲存在伺服器（留言板）
- DOM-based XSS：客戶端 JavaScript 直接操作 DOM
- 防禦：輸出編碼（Output Encoding）、Content Security Policy（CSP）、HttpOnly Cookie

**跨站請求偽造（CSRF, Cross-Site Request Forgery）**
- 誘使已登入使用者的瀏覽器發送惡意請求
- 防禦：CSRF Token、SameSite Cookie 屬性、Referer 檢查

**目錄遍歷（Path Traversal）**
- 使用 `../` 跳脫 Web 根目錄存取系統檔案
- 防禦：輸入驗證、白名單路徑、chroot jail

**XML 外部實體（XXE, XML External Entity）**
- 利用 XML 解析器處理外部實體讀取本地檔案或 SSRF
- 防禦：停用外部實體、使用安全的 XML 解析器設定

**指令注入（Command Injection）**
- 將系統指令插入應用程式輸入，在伺服器執行任意命令
- 防禦：避免直接呼叫 shell、輸入驗證與跳脫

#### 社交工程與釣魚

| 類型 | 說明 |
|---|---|
| 釣魚（Phishing） | 大量仿冒電子郵件誘使點擊惡意連結 |
| 魚叉式釣魚（Spear Phishing） | 針對特定目標的精準釣魚 |
| 捕鯨（Whaling） | 鎖定高階主管的釣魚攻擊 |
| 語音釣魚（Vishing） | 電話假冒客服誘騙資訊 |
| 簡訊釣魚（Smishing） | 簡訊釣魚 |
| 誘餌（Baiting） | 提供受感染的 USB 或下載 |
| 假冒（Pretexting） | 編造情境取得信任以騙取資訊 |

**防禦**：資安意識訓練、郵件閘道過濾（SPF/DKIM/DMARC）、MFA

#### 惡意程式分類

| 類型 | 特徵 |
|---|---|
| 病毒（Virus） | 附著在合法程式，需執行宿主才傳播 |
| 蠕蟲（Worm） | 自我複製傳播，不需宿主 |
| 木馬（Trojan） | 偽裝成合法程式，暗中執行惡意功能 |
| 勒索軟體（Ransomware） | 加密檔案勒索贖金 |
| 間諜軟體（Spyware） | 竊取使用者資訊 |
| 後門（Backdoor） | 繞過正常認證的秘密入口 |
| Rootkit | 隱藏惡意程式存在，通常在核心層運作 |
| 殭屍網路（Botnet） | 受控的受感染主機群（Bot），受 C&C 伺服器指揮 |
| APT（進階持續性威脅） | 有組織、長期潛伏的目標式攻擊 |

---

#### 攻擊鏈（Cyber Kill Chain / ATT&CK）

**Lockheed Martin 網路殺傷鏈（7 步驟）**
1. 偵察（Reconnaissance）
2. 武器化（Weaponization）
3. 傳遞（Delivery）
4. 漏洞利用（Exploitation）
5. 安裝（Installation）
6. 命令與控制（C&C）
7. 目標達成（Actions on Objectives）

**MITRE ATT&CK**
- 描述真實攻擊者戰術（Tactics）、技術（Techniques）、程序（Procedures）的知識庫
- 用途：威脅情報分析、紅隊演練、偵測規則設計

### 1-2 防護與應變實務

#### 網路安全防禦設備

| 設備 | 說明 |
|---|---|
| 防火牆（Firewall） | 依規則過濾封包（L3/L4） |
| 新世代防火牆（NGFW） | 加入應用層識別、IPS、SSL 檢查 |
| IDS（入侵偵測系統） | 偵測並告警，不主動阻擋 |
| IPS（入侵防禦系統） | 偵測並主動阻擋惡意流量 |
| WAF（Web 應用防火牆） | 保護 Web 應用，過濾 SQLi/XSS 等 |
| DLP（資料外洩防護） | 偵測並阻止敏感資料外洩 |
| EDR（端點偵測與回應） | 端點威脅偵測、調查、回應；行為分析、記憶體注入偵測、即時隔離 |
| NDR（網路偵測與回應） | 網路流量分析、橫向移動偵測（被動監聽，不影響生產） |
| XDR（延伸偵測與回應） | 整合 EDR+NDR+身分的跨來源關聯分析平台 |
| MDR（委外偵測與回應） | 外部 24×7 監控與即時回應服務，補足夜間/假日人力空窗 |
| SIEM | 集中收集日誌，關聯分析、告警 |
| SOAR | 安全協調、自動化與回應平台 |
| Honeypot | 誘捕系統，收集攻擊者行為情報 |
| BAS（入侵與攻擊模擬） | 持續高頻率模擬攻擊，驗證現有防禦機制是否有效 |
| EASM（外部攻擊面管理） | 查核企業對外暴露的數位資產：開放 Port、網域、外洩帳密等 |
| CSPM（雲端安全態勢管理） | 持續監控並自動修正雲端設定錯誤（如 S3 bucket 開放存取） |

#### 事件回應（Incident Response, IR）流程
依 NIST SP 800-61 標準：

1. **準備（Preparation）**：建立 IR 計畫、組成 CSIRT、準備工具
2. **偵測與分析（Detection & Analysis）**：識別事件、蒐集 IoC、確認嚴重度
3. **圍堵（Containment）**：
   - 短期圍堵：隔離受感染系統
   - 長期圍堵：部署補丁、修改設定
4. **根除（Eradication）**：清除惡意程式、關閉後門、修補弱點
5. **復原（Recovery）**：系統還原、服務重啟、監控驗證
6. **事後學習（Post-Incident Activity）**：撰寫事件報告、改善流程

#### 數位鑑識（Digital Forensics）基本原則
- **保全證據（Evidence Preservation）**：先製作映像檔（Disk Image），對原始媒體只讀不寫
- **監管鏈（Chain of Custody）**：完整記錄證據存取歷程
- **雜湊驗證**：使用 MD5/SHA-256 驗證映像完整性
- **揮發性資料順序（Volatility Order）**：CPU 暫存器 → 記憶體 → 網路連線 → 磁碟 → 備份

#### IoC（威脅指標）類型
- **檔案型**：雜湊值（MD5/SHA256）、惡意檔案名稱
- **網路型**：惡意 IP、網域、URL
- **行為型**：特定登錄檔修改、異常網路連線
- STIX / TAXII：威脅情報共享標準格式

## 評鑑主題二：作業安全實務

### 2-1 安全維運

#### SIEM（Security Information and Event Management）
- 功能：日誌蒐集、正規化、關聯分析、告警、報表
- 關鍵概念：
  - **Use Case**：設計偵測規則的情境（如多次失敗登入）
  - **UEBA（使用者與實體行為分析）**：基線行為 + 異常偵測
  - **日誌來源**：防火牆、IPS、AD、Web Server、EDR

#### 弱點管理（Vulnerability Management）生命週期
1. **資產盤點**：識別所有 IT 資產
2. **弱點掃描**：工具（Nessus、OpenVAS、Qualys）定期掃描
3. **風險評估**：依 CVSS 分數與資產重要性排序
4. **修補（Patching）**：依優先順序修補
5. **驗證**：修補後重新掃描確認
6. **報告**：追蹤修補率與弱點趨勢

#### 修補管理（Patch Management）
- **緊急修補（Emergency Patch）**：高嚴重度（CVSS 9+），24–72 小時內完成
- **定期修補**：每月「補丁星期二」（Patch Tuesday，微軟第二個周二）
- **修補測試**：先在測試環境驗證，再推正式環境
- **補償控制**：無法立即修補時，使用 WAF 規則或網路隔離作為臨時措施

#### 日誌管理
- **日誌類型**：系統日誌、應用日誌、安全日誌、網路流量日誌
- **保留原則**：法規要求（如資安法要求 6 個月以上），高風險系統建議 1 年以上
- **NTP 時間同步**：所有設備統一時間源，確保日誌可關聯分析
- **集中式日誌伺服器（Syslog）**：避免攻擊者在本機篡改日誌
- **日誌完整性**：使用雜湊或數位簽章保護日誌不被竄改

#### 資安監控（SOC 運作）
- **SOC（安全運營中心）**：24×7 監控、告警分類、事件回應
- **告警等級分類（Triage）**：
  - L1：初步分類，判斷真假陽性
  - L2：深入調查分析
  - L3：複雜事件處理、威脅獵捕
- **MTTR（平均修復時間）** 與 **MTTD（平均偵測時間）**：衡量 SOC 效率的關鍵指標

#### 加密管理
- **對稱加密**：AES-256（速度快，適合大量資料）；需安全金鑰交換
- **非對稱加密**：RSA-2048 / ECC；用於金鑰交換與數位簽章
- **雜湊函數**：SHA-256、SHA-3（不可逆，用於完整性驗證）
- **PKI（公鑰基礎建設）**：CA 核發憑證、CRL/OCSP 撤銷機制
- **TLS 1.3**：目前建議版本，禁用 SSLv3、TLS 1.0/1.1

### 2-2 滲透測試、源碼檢測、資安健診

#### 滲透測試（Penetration Testing）

**測試類型**
| 類型 | 知識程度 | 說明 |
|---|---|---|
| 黑箱（Black Box） | 零知識 | 模擬外部攻擊者 |
| 白箱（White Box） | 完全知識 | 提供原始碼、架構圖、帳號 |
| 灰箱（Grey Box） | 部分知識 | 介於兩者之間，模擬內鬼 |

**測試範圍**
- 網路滲透測試（外部/內部）
- Web 應用程式滲透測試
- 行動應用程式測試
- 社交工程測試
- 紅隊演練（Red Team Exercise）：全方位模擬 APT 攻擊

**滲透測試流程（依 PTES 標準）**
1. **事前溝通（Pre-engagement）**：確定範圍、法律授權、時間
2. **情報蒐集（Reconnaissance）**：OSINT、主動/被動偵查
3. **威脅建模（Threat Modeling）**：識別目標與攻擊路徑
4. **弱點分析（Vulnerability Analysis）**：掃描與手動測試
5. **漏洞利用（Exploitation）**：嘗試取得存取權限
6. **後滲透（Post-Exploitation）**：權限提升、橫向移動、持久化
7. **報告撰寫（Reporting）**：執行摘要 + 技術細節 + 建議

**常用工具**
| 類別 | 工具 |
|---|---|
| 網路掃描 | Nmap、Masscan |
| 弱點掃描 | Nessus、OpenVAS |
| Web 測試 | Burp Suite、OWASP ZAP |
| 漏洞利用框架 | Metasploit |
| 密碼破解 | Hashcat、John the Ripper |
| 流量分析 | Wireshark |
| OSINT | Maltego、Shodan、theHarvester |

---

#### 源碼檢測（Source Code Review / SAST）

**靜態應用程式安全測試（SAST）**
- 在不執行程式的情況下，分析原始碼、位元組碼或二進位碼
- 優點：早期發現問題（Shift Left）、可掃描 100% 代碼
- 缺點：誤報率高、無法發現執行時期問題

**動態應用程式安全測試（DAST）**
- 對執行中的應用程式進行黑箱測試
- 優點：發現執行時期問題、誤報率低
- 缺點：覆蓋率受測試路徑影響

**互動式應用程式安全測試（IAST）**
- 結合 SAST + DAST，在應用執行中插入探針監控

**SCA（軟體組成分析）**
- 掃描第三方套件與開源元件的已知 CVE
- 工具：Snyk、OWASP Dependency-Check、Black Duck

**安全開發生命週期（SDL / S-SDLC）**
| 階段 | 資安活動 |
|---|---|
| 需求 | 安全需求、威脅模型 |
| 設計 | 安全架構審查、威脅建模（STRIDE） |
| 開發 | 安全編碼標準、SAST |
| 測試 | DAST、滲透測試 |
| 部署 | 組態強化、安全掃描 |
| 維護 | 弱點管理、修補 |

**STRIDE 威脅模型**
| 字母 | 威脅 | 破壞的安全屬性 |
|---|---|---|
| S | Spoofing（偽冒） | 身份驗證 |
| T | Tampering（竄改） | 完整性 |
| R | Repudiation（否認） | 不可否認性 |
| I | Information Disclosure（資訊洩露） | 機密性 |
| D | Denial of Service（阻斷服務） | 可用性 |
| E | Elevation of Privilege（權限提升） | 授權 |

---

#### 資安健診（Security Health Check）

**定義**
資安健診為政府機關及企業定期進行的資安狀態評估，依據台灣資安法規，A 級機關每年至少進行一次資安健診。

**健診項目（依政府資安健診規範）**
| 項目 | 說明 |
|---|---|
| 網路架構檢視 | 確認網路拓樸、防火牆規則合理性 |
| 使用者端電腦檢視 | 作業系統修補、防毒更新狀況 |
| 伺服器主機檢視 | 服務清單、帳號管理、修補狀況 |
| 網路惡意活動檢視 | 對外連線分析、異常流量偵測 |
| 資安設備檢視 | 防火牆、IPS 組態與規則審查 |

**資安成熟度評估**
- **CMMI（能力成熟度整合模型）**：1（初始）→ 2（已管理）→ 3（已定義）→ 4（量化管理）→ 5（最佳化）
- **SSE-CMM**：專注資訊安全工程的成熟度模型

**與滲透測試的差異**
| 項目 | 資安健診 | 滲透測試 |
|---|---|---|
| 目的 | 全面評估資安狀態 | 模擬攻擊找出可利用弱點 |
| 深度 | 廣度優先 | 深度優先 |
| 授權 | 通常為法規要求 | 合約授權 |
| 結果 | 合規狀態報告 | 弱點利用技術報告 |

## 補充主題二：特殊攻擊手法補充

### Slowloris 攻擊
- 類型：應用層 DoS
- 原理：建立大量不完整的 HTTP 連線，保持連線開啟耗盡 Web 伺服器連線資源
- 防禦：設定連線逾時、限制每 IP 連線數、使用反向代理

### BlueBorne 攻擊
- 透過**藍牙（Bluetooth）**漏洞遠端攻擊，不需配對即可執行任意程式碼
- 限制：需在藍牙訊號範圍內，非遠端網路攻擊

### Evil Twin 攻擊
- 架設與合法 Wi-Fi 同名的惡意熱點，誘騙使用者連線進行 MITM
- 防禦：VPN、802.1X 企業認證、不使用公共 Wi-Fi

### 攻擊手法遠端可達性比較（考試重點）
| 攻擊手法 | 可遠端攻擊網路伺服器？ |
|---|---|
| Slowloris（應用層 DoS） | ✅ 是 |
| SQL Injection | ✅ 是 |
| BlueBorne | ❌ 否（需藍牙範圍） |
| Evil Twin | ❌ 否（需同場域 Wi-Fi） |

## 補充主題三：電子郵件安全鑑識

### 郵件驗證機制
| 機制 | 說明 | 防禦的攻擊 |
|---|---|---|
| **SPF** | 發布授權 IP 清單，防止他人冒用網域發信 | 寄件者偽造 |
| **DKIM** | 寄件伺服器對郵件標頭加數位簽章 | 郵件竄改、寄件者偽造 |
| **DMARC** | 整合 SPF/DKIM，定義驗證失敗時的處理政策（none/quarantine/reject） | 品牌仿冒 |

### DKIM 簽名分析
- `DKIM-Signature: No signature`：郵件未經 DKIM 簽名，不代表一定惡意，但應驗證來源可疑性。
- 正確處置：**驗證信件來源，檢查其他可疑跡象**（非自動信任，也非立即刪除）。

### Email Header 分析重點
- `Received: from 10.0.1.7`（本地 IP）：攻擊者可能透過被入侵的內部主機中繼，隱藏實際來源。
- `X-DKIM-ATPS`、`X-Spam-Status`：垃圾郵件過濾結果。
- 分析工具：VirusTotal（URL 與附件）、MXToolbox、Email Header Analyzer。

## 補充主題四：日誌不可否認性與完整性

### 確保日誌不可否認性的最佳實務
- **Syslog 即時拋送至遠端專用伺服器** + 數位簽章或雜湊鏈結（Hash Chain）
- 好處：攻擊者即使清除本地日誌，遠端仍有備份；簽章確保完整性
- ❌ 僅本地儲存（攻擊者可篡改）
- ❌ 設定日誌滿額自動覆蓋（破壞可用性）

### 日誌完整性方法
- **數位簽章（Digital Signature）**：每條日誌加簽，確保不可否認性
- **雜湊鏈結（Hash Chain）**：每條日誌含前一條的雜湊值，任何修改可被發現
- **集中式 SIEM**：集中收集，難以單點竄改

## 補充主題五：勒索軟體防禦與安全復原

### 防範勒索軟體最有效措施
- **離線備份（Offline Backup / Air-gapped Backup）**：勒索軟體最難加密的備份形式
- 定期資安意識訓練與社交工程演練
- 定期弱點掃描
- 資安保險（轉移財務風險）
- ❌ 僅有線上備份（Online）：勒索軟體通常也會加密網路共享備份

### 3-2-1 備份原則
- **3** 份備份、**2** 種不同媒體、**1** 份異地離線保存

### 安全復原（BCP）必要步驟
1. **隔離受感染網段** + 驗證備份檔完整性（雜湊比對）
2. 找出入侵破口（Root Cause）並完成修補
3. **重設所有特權帳號密碼與憑證**（攻擊者可能已竊取）
4. 確認系統安全後才重新上線
5. ❌ 不建議立即支付贖金（解密不保證成功，且助長犯罪）

## 補充主題六：弱點優先排序策略

### 正確優先順序（考試重點）
**優先修補「對外暴露且可被實際利用」的高風險弱點**，再依資產重要性與可利用性排序。

| 因素 | 說明 |
|---|---|
| **暴露面（Exposure）** | 是否對外開放、是否有防火牆保護 |
| **可利用性（Exploitability）** | 是否有公開 PoC 或已被積極利用 |
| **資產重要性** | 核心業務系統優先 |
| CVSS 分數 | 參考依據，但不能單純按分數排序 |

❌ 不應：按 CVSS 純順序修、先修最多數量的同類弱點、先修最容易修的弱點。

## 補充主題七：滲透測試實戰技術

### SQL Injection 測試語法

| 語法 | 用途 | 注意 |
|---|---|---|
| `OR '1'='1' --` | 測試基本 SQLi 是否存在 | 常用，但 DROP TABLE 等危險語法實戰不使用 |
| `'; IF (1=1) WAITFOR DELAY '0:0:10'--` | Time-based Blind SQLi | 會讓伺服器延遲 10 秒，謹慎使用 |
| `'; DROP TABLE Products;#` | 刪除資料表 | 實戰禁用，破壞性極大 |
| `INTO OUTFILE` | 寫入 Webshell | 特定情境可達成 RCE |

**Blind SQL Injection（盲注）**：
- 無法透過錯誤訊息取得資訊
- 透過行為模式（時間延遲、Boolean 回應）推斷資料
- 可手動執行（用 time-based 或 boolean-based）
- 工具：sqlmap

**防範 SQL Injection（最有效）**：
- ✅ 參數化查詢（Prepared Statements / Parameterized Query）
- ✅ 正則表示式過濾輸入
- ✅ 最小權限資料庫帳號、關閉錯誤訊息
- ❌ 限制輸入字元長度（僅限制長度無法防止注入）

### Broken Access Control / IDOR
- **IDOR（Insecure Direct Object Reference）**：攻擊者修改 API 參數（如 `orderId=10025` 改為其他值）存取其他使用者資料
- OWASP A01（2021）排名第一
- 防禦：後端驗證存取權限、**將參數雜湊加鹽隱碼處理**（使外部無法猜測內部 ID）

### WAF 繞過技術（常見手法）
- **Encoding 混淆**：URL Encoding、Unicode 編碼讓 WAF 無法識別惡意 Payload
- **Payload 關鍵字替換**：將 `OR` 改用 `||`，繞過黑名單規則
- ❌ **無效方法**：
  - 變換 IP/Proxy（WAF 通常不基於 IP 判斷 SQLi 語法）
  - ARP Spoofing（不能用來繞過 WAF，WAF 在應用層）

### JWT（JSON Web Token）安全

**JWT 結構**：`Header.Payload.Signature`（三段 Base64Url 編碼）

| 演算法 | 類型 | 說明 |
|---|---|---|
| HS256 | 對稱（HMAC+SHA-256） | 同一金鑰簽發與驗證，適合單一服務 |
| RS256 | 非對稱（RSA+SHA-256） | 私鑰簽發，公鑰驗證，適合多服務 |
| ES256 | 非對稱（ECDSA） | 更輕量的非對稱簽章 |

**JWT 安全弱點**：
- `alg: none` 攻擊：將演算法設為 none，伺服器若不檢查則接受任何 token
- 弱 Secret Key：可被暴力破解
- 防禦：驗證 alg 欄位、使用強隨機 Secret、設定 Token 過期時間

### 行動應用程式安全：混淆（Obfuscation）
- 目的：提高逆向工程的難度（不能完全防止，只能增加困難度）
- 方式：程式碼轉換、變數重命名、控制流混淆、字串加密
- 工具（逆向）：jadx、apktool、Frida
- ❌ 不會加速執行速度或減少檔案大小（可能反而更大）
