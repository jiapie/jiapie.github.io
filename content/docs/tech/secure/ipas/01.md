## 評鑑主題一：資訊安全管理系統

### 1-1 ISMS 建置實務與法規遵循

#### ISO/IEC 27001 架構
- **ISMS（Information Security Management System）**：依 ISO 27001 建立、實作、維護、持續改善資訊安全管理系統。
- PDCA 循環：Plan（規劃）→ Do（執行）→ Check（查核）→ Act（改善）
- 核心條款（Clause 4–10）：
  - Clause 4：組織情境（內外部議題、利害關係人需求）
  - Clause 5：領導（政策、角色職責）
  - Clause 6：規劃（風險處理、資安目標）
  - Clause 7：支援（資源、能力、意識、文件化資訊）
  - Clause 8：運作（風險評鑑與處理執行）
  - Clause 9：績效評估（內部稽核、管理審查）
  - Clause 10：改善（不符合事項、矯正措施）
- **附錄 A（Annex A）**：93 項控制措施，分為 4 大主題：
  - 組織控制（Organizational controls）
  - 人員控制（People controls）
  - 實體控制（Physical controls）
  - 技術控制（Technological controls）

#### 相關法規遵循（台灣）
- **資通安全管理法**（資安法）：適用政府機關與關鍵基礎設施，規定資安長、資安專責人員、資安計畫、稽核等義務。
- **資通安全責任等級分級辦法**：A～E 級，依機關規模與業務敏感度分級，規定各級應遵循的控制措施。
- **個人資料保護法（個資法）**：規範個資蒐集、處理、利用，要求資安防護措施，違規最高罰款 50 萬元（民事損害賠償另計）。
- **電子簽章法**：規範數位憑證與電子簽章的法律效力。
- **GDPR**（歐盟）：適用處理歐盟公民資料的企業，要求 72 小時資料外洩通報、資料保護官（DPO）等。

#### 文件化資訊（Documented Information）
| 必要文件 | 說明 |
|---|---|
| 資訊安全政策 | 最高層級承諾聲明 |
| 資訊安全目標 | 可量化的安全目標 |
| 風險評鑑報告 | 含方法論、結果 |
| 風險處理計畫（RTP） | 控制措施選擇與殘餘風險 |
| 適用性聲明（SoA） | 所有 Annex A 控制措施的選擇/排除說明 |
| 內部稽核報告 | 週期性執行 |
| 管理審查記錄 | 最高管理層定期審查 |

### 1-2 資安管理實務：存取控制、縱深防禦、職務區隔（SOD）

#### 存取控制（Access Control）
- **最小權限原則（Principle of Least Privilege, PoLP）**：使用者/系統只授予完成工作所需的最小權限。
- **需知原則（Need-to-Know）**：資訊只分享給需要知道的人。
- **存取控制模型**：

| 模型 | 說明 | 範例 |
|---|---|---|
| DAC（自主存取控制） | 資源擁有者決定存取權限 | Linux 檔案權限 |
| MAC（強制存取控制） | 系統依標籤強制管控 | 機密等級（絕密/機密/一般） |
| RBAC（角色型存取控制） | 依角色授予權限 | 企業 ERP 系統 |
| ABAC（屬性型存取控制） | 依屬性（時間、地點等）動態授權 | 零信任架構 |

- **身份驗證因子**：
  - 知識因子（you know）：密碼、PIN
  - 持有因子（you have）：OTP 裝置、智慧卡
  - 生物因子（you are）：指紋、臉部辨識
  - **MFA（多因子驗證）**：結合 2 種以上因子

- **特權帳號管理（PAM, Privileged Access Management）**：集中管理管理員帳號，記錄所有操作，定期輪換密碼。

#### 縱深防禦（Defense in Depth）
- 概念：採用多層次安全控制，單一控制失效不致全面失守。
- 層次架構（由外而內）：
  1. 實體安全（門禁、監控）
  2. 網路邊界（防火牆、IPS）
  3. 網路分區（VLAN、DMZ）
  4. 主機安全（作業系統強化、防毒）
  5. 應用程式安全（WAF、SAST/DAST）
  6. 資料安全（加密、DLP）
  7. 人員（資安意識訓練）

- **DMZ（非軍事區）**：將對外服務（Web、Mail）放置在 DMZ，與內部網路隔離，減少暴露面。

#### 職務區隔（Segregation of Duties, SOD）
- 核心概念：一件高風險操作需要至少兩個不同人員參與，防止單人舞弊。
- 典型應用：
  - IT：程式開發人員不得同時擁有正式環境部署權限
  - 財務：請款人員與核款人員分離
  - 資安：資安政策制定者與稽核者分離
- **雙人完整性（Two-Person Integrity）**：特定高風險操作需兩人同時在場。
- **輪調（Job Rotation）**：定期輪換職務，降低長期舞弊風險，也有助於發現異常。

### 1-3 安全架構規劃實務

#### 零信任架構（Zero Trust Architecture, ZTA）
- 核心原則：「永不信任，持續驗證（Never trust, always verify）」
- 3 大支柱：
  1. **身份驗證**：強 MFA，持續驗證使用者與裝置
  2. **最小權限**：依情境動態授權
  3. **微分段（Micro-segmentation）**：最小化橫向移動
- NIST SP 800-207：零信任架構標準參考文件

#### 安全架構框架
| 框架 | 用途 |
|---|---|
| TOGAF | 企業架構整體規劃 |
| SABSA | 資安架構規劃（從業務需求到技術控制） |
| NIST CSF | 資安能力成熟度框架（識別/保護/偵測/回應/復原） |
| CIS Controls | 18 項優先實施的資安控制 |

#### NIST 網路安全框架（CSF）5 大功能
1. **識別（Identify）**：資產管理、風險評估、治理
2. **保護（Protect）**：存取控制、資安意識、資料安全、維護
3. **偵測（Detect）**：持續監控、異常與事件偵測
4. **回應（Respond）**：事件回應規劃、通訊、分析、緩解
5. **復原（Recover）**：復原規劃、改善、溝通

#### 網路分區設計
```
Internet
   │
[防火牆 L1]
   │
  DMZ（Web Server、Mail Server、DNS）
   │
[防火牆 L2]
   │
內部網路
 ├─ 辦公室 VLAN
 ├─ 伺服器 VLAN
 └─ 管理 VLAN（隔離管理流量）
```

#### 雲端安全架構
- **共同責任模型（Shared Responsibility Model）**：
  - IaaS：CSP 負責實體/網路，用戶負責 OS/應用/資料
  - PaaS：CSP 負責至平台層，用戶負責應用/資料
  - SaaS：CSP 負責絕大部分，用戶只負責資料與存取管理
- **CASB（Cloud Access Security Broker）**：監控並控制雲端服務使用。
- **CSPM（Cloud Security Posture Management）**：自動化偵測雲端錯誤設定。

## 評鑑主題二：風險管理實務

### 2-1 風險分析與評估

#### 基本概念
| 術語 | 定義 |
|---|---|
| 資產（Asset） | 具有價值需保護的事物（資料、系統、人員） |
| 威脅（Threat） | 可能造成傷害的事件或行為者 |
| 弱點（Vulnerability） | 可被威脅利用的缺陷 |
| 風險（Risk） | 威脅利用弱點對資產造成衝擊的可能性與影響 |
| 衝擊（Impact） | 安全事件發生後的後果 |
| 可能性（Likelihood） | 威脅發生的機率 |

**風險 = 威脅 × 弱點 × 衝擊**

#### 風險評估方法

**定性風險評估（Qualitative）**
- 使用高/中/低或 1–5 分尺度
- 優點：簡單快速，無需精確數據
- 缺點：主觀性高

**定量風險評估（Quantitative）**
- **SLE（單一損失預期）** = 資產價值 × 暴露因子（EF）
- **ALE（年度損失預期）** = SLE × ARO（年度發生率）
- 優點：可與安全投資進行成本效益分析
- 缺點：需要可靠的歷史數據

**CVSS（通用弱點評分系統）**
- 0.0–10.0 分
- 三個評分群組：基本（Base）/ 時間（Temporal）/ 環境（Environmental）
- 基本向量：攻擊向量、攻擊複雜度、所需權限、使用者互動、範圍、機密性/完整性/可用性影響

#### 風險評估標準與方法論
| 方法 | 說明 |
|---|---|
| ISO 31000 | 風險管理通用原則與指引 |
| ISO/IEC 27005 | 資訊安全風險管理 |
| OCTAVE | 組織導向的威脅、資產、弱點評估 |
| FAIR | 因素分析資訊風險（定量方法） |

#### 資產分類分級
- 分類依據：機密性（Confidentiality）、完整性（Integrity）、可用性（Availability）
- 台灣政府分級：絕密、機密、密、一般（依國家機密保護法）
- 企業常用：高/中/低 三級，定義存取控制、加密、備份等對應要求

### 2-2 風險處理實務

#### 四種風險處理策略（4T）
| 策略 | 說明 | 範例 |
|---|---|---|
| **規避（Avoid）** | 停止或不從事該風險活動 | 不開發高風險功能、停止使用有漏洞的系統 |
| **降低/緩解（Reduce/Mitigate）** | 實施控制措施降低可能性或衝擊 | 安裝防火牆、修補程式、加密資料 |
| **轉移（Transfer）** | 將風險轉移給第三方 | 採購網路資安保險、委外服務 |
| **接受（Accept）** | 知情接受殘餘風險 | 低風險且控制成本高於損失時 |

#### 控制措施類型
- 依目的分：
  - **預防性（Preventive）**：防止事件發生（存取控制、加密）
  - **偵測性（Detective）**：發現事件已發生（SIEM、IDS、稽核日誌）
  - **矯正性（Corrective）**：事後修復（備份還原、漏洞修補）
  - **嚇阻性（Deterrent）**：降低攻擊意願（警告標語、法律威脅）
  - **補償性（Compensating）**：無法實施主要控制時的替代措施

- 依性質分：
  - **行政/管理（Administrative）**：政策、程序、訓練
  - **技術（Technical）**：防火牆、加密、MFA
  - **實體（Physical）**：門禁、監視器、機房管制

#### 風險處理計畫（Risk Treatment Plan, RTP）
必要內容：
- 所選控制措施
- 預期實施日期與負責人
- 所需資源
- 實施後的殘餘風險（Residual Risk）
- 管理層接受殘餘風險的簽核

#### 殘餘風險（Residual Risk）
- 實施控制後仍存在的剩餘風險
- 必須由管理層明確接受（簽核）
- 需持續監控，不代表可忽視

#### BCP / DRP
| 項目 | BCP | DRP |
|---|---|---|
| 全名 | 營運持續計畫 | 災害復原計畫 |
| 範疇 | 整體業務持續 | IT 系統復原 |
| 重點 | 如何在中斷期間維持業務 | 如何恢復 IT 系統 |

- **RTO（復原時間目標）**：系統必須在多長時間內恢復
- **RPO（復原點目標）**：可接受的最大資料損失時間
- **BIA（業務衝擊分析）**：識別關鍵業務流程與最大可接受中斷時間（MTPD）

## 補充主題一：後量子密碼學（PQC）與加密敏捷性

### 威脅背景
- **「先竊取，後解密（Harvest Now, Decrypt Later）」**：攻擊者現在蒐集加密資料，待量子電腦成熟後再解密。對需保存 10 年以上的機密資料構成威脅。
- 現有公鑰體系（RSA、ECC）將在量子電腦時代失效。

### NIST PQC 標準（2024 正式發布）
| 演算法 | 用途 |
|---|---|
| ML-KEM（Kyber）| 金鑰封裝（Key Encapsulation） |
| ML-DSA（Dilithium）| 數位簽章 |
| SLH-DSA（SPHINCS+）| 數位簽章（備用） |
| FN-DSA（Falcon）| 數位簽章 |

### 加密敏捷性（Crypto-Agility）
- 設計系統時使演算法可快速替換，無需大幅重構。
- **混合模式（Hybrid Scheme）**：同時使用傳統演算法（ECC/RSA）與後量子演算法雙重封裝，兩者均需被破解才能洩密。此為當前最佳實務。
- ❌ 錯誤做法：僅升級 RSA 金鑰長度（RSA-4096 仍可被量子電腦破解）。

## 補充主題二：台灣上市公司重大資安事件通報

### 臺灣證券交易所規定
上市公司發生下列事件須發布重大訊息：
- 官方網站首頁遭置換（Defacement）
- ERP/核心系統遭勒索軟體加密導致停擺
- 網路遭入侵、內部文件資料外洩
- 機密檔案遭勒索軟體加密
- 遭受 DDoS 攻擊（服務中斷）

**不須發布**的情況（參考）：雲端備份供應商故障但主線業務正常、備份資料未遺失亦未外洩。

### 資通安全管理法通報規定
- 發生資安事件應於知悉後**法定時限**內通報中央目的事業主管機關。
- 個資外洩亦屬資安事件，須納入資通安全事件通報及應變辦法。
- ISO 27001 認證不能免除個資法的行政罰鍰與民事賠償責任。

### 受委託業務安全性檢測
- 委託金額達一定規模的資通系統，**不得**由承攬廠商自行出具安全性證明。
- 合法方式：委託廠商提供第三方安全性檢測、客戶自行檢測、客戶另委第三方檢測。

## 補充主題三：生物特徵認證（Biometric Authentication）

### Type I / Type II 錯誤
| 術語 | 別名 | 說明 |
|---|---|---|
| **FAR（錯誤接受率）** | Type II 錯誤 | 冒用者被錯誤接受（安全性問題） |
| **FRR（錯誤拒絕率）** | Type I 錯誤 | 合法使用者被錯誤拒絕（可用性問題） |
| **CER（交叉錯誤率）** | EER | FAR = FRR 的交叉點，用來比較系統準確度 |

> ⚠️ 考試重點：**錯誤拒絕率（FRR）= Type I 錯誤**，錯誤接受率（FAR）= Type II 錯誤。

### 各種生物特徵比較
| 技術 | 準確度 | 特點 |
|---|---|---|
| 指紋辨識 | 高 | 普及、便利 |
| 視網膜辨識 | 極高 | 難以偽造，但取樣不便，難以普及 |
| 虹膜辨識 | 極高 | 比視網膜方便，仍較昂貴 |
| 臉部辨識 | 中-高 | 易受光線、角度影響，Deepfake 可偽造 |
| 聲紋辨識 | 中 | 易受環境雜音影響 |

### FIDO2 / WebAuthn / Passkey
- 基於公鑰加密的無密碼認證標準。
- Passkey：裝置端產生金鑰對，私鑰不離開裝置，安全性高於 SMS OTP。
- 最能抵抗釣魚攻擊的認證方式之一。

## 補充主題四：身分治理與管理（IGA）

**IGA（Identity Governance and Administration）** 核心功能：

| 功能 | 說明 |
|---|---|
| **存取認證（Access Certification/Review）** | 定期觸發權限盤點，要求主管確認員工權限的必要性 |
| **權限生命週期管理（Entitlement Management）** | 人員到職/轉調/離職時自動開通或回收權限 |
| **角色挖掘（Role Mining）** | 分析現有權限分配，優化角色設計（不是維持現況） |
| **職務區隔分析（SOD Analysis）** | 偵測衝突的權限組合 |
| **特權帳號管理（PAM）** | 管理管理員帳號，記錄存取行為 |

**ABAC 屬性類型（考試重點）**：
- **主體屬性（Subject）**：使用者職稱、部門
- **資源屬性（Resource）**：資料機密等級標籤
- **環境屬性（Environment）**：存取時間、來源地理位置（考試答案）
- **動作屬性（Action）**：讀取、修改、刪除

## 補充主題五：SBOM 與軟體供應鏈安全

### SBOM（Software Bill of Materials，軟體物料清單）
- 列出軟體所有元件、相依套件、版本的清單。
- 當新 CVE 揭露時，可即時定位受影響系統（如 Log4j 事件）。
- 應整合進 CI/CD 流程，自動更新與掃描。

### VEX（Vulnerability Exploitability eXchange）
- 配合 SBOM 使用，標記「雖存在於元件中但實際不可被利用」的漏洞。
- 能有效過濾誤報，縮短漏洞響應時間。

### DevSecOps 整合
```
開發 → SAST（靜態分析）
建置 → SCA（第三方元件掃描）+ SBOM 生成
測試 → DAST（動態測試）
部署 → CSPM（雲端設定稽核）
維運 → 弱點管理 + VEX 過濾
```

## 補充主題六：CISA 零信任成熟度模型 2.0

**五大支柱（2023 年 4 月）**：
1. **身份（Identity）**：使用者、服務帳號、裝置身份管理
2. **裝置（Devices）**：端點健康度驗證
3. **網路（Networks）**：微分段、加密傳輸
4. **應用程式與工作負載（Applications & Workloads）**：應用層存取控制
5. **資料（Data）**：資料分類、加密、DLP

> ❌ 「稽核」不是五支柱之一（考試陷阱）。

### ZTNA vs 傳統 VPN
| 項目 | 傳統 VPN | ZTNA |
|---|---|---|
| 連線範圍 | 連線後可存取全網段 | 僅建立應用程式層級通道（App-level tunnel） |
| 授權邏輯 | 帳號密碼 + IP 白名單 | 身份 + 裝置健康度 + 情境（動態授權） |
| 橫向移動風險 | 高 | 低 |

## 補充主題七：OT / ICS 資訊安全

### 普渡模型（Purdue Model）
```
Level 5: 企業網路（ERP、Email）
Level 4: 工廠業務規劃
─────────── IT/OT 邊界（IDMZ）───────────
Level 3: 工廠/產線管理（MES、SCADA）
Level 2: 監控層（HMI、工程師站）
Level 1: 控制層（PLC、DCS）
Level 0: 現場設備（感測器、執行器）
```

### IEC 62443 工控資安標準
- Zone（安全區域）：依安全等級分組管理資產
- Conduit（通道）：連接不同 Zone 的通訊路徑，需設置防火牆（Conduit Firewall）
- **IDMZ（工控 DMZ）**：在 IT 與 OT 之間建立中間緩衝區，終止直接連線 → **最佳實務**
- 安全等級（Security Level, SL）：SL1（低）到 SL4（高）

### OT vs IT 事件應變優先順序
| 優先考量 | IT | OT |
|---|---|---|
| 第一優先 | 機密性（Confidentiality） | **人員安全與可用性** |
| 原因 | 資料保護 | 停線影響人身安全與生產 |

### 虛擬修補（Virtual Patching）
- 對無法直接安裝修補程式的老舊 PLC 設備使用。
- 實現方式：**工控 IPS/Firewall（OT IPS）** 在網路層阻擋針對已知弱點的攻擊。
- ❌ 非端點防毒、非 WAF。

### OT 環境可視性提升（不影響產線）
- **Port Mirroring / SPAN**：複製流量給偵測設備（被動分析，不影響生產）
- **Network TAP**：擷取實體線路訊號（被動）
- **SCADA 主機輕量代理程式**：收集日誌
- ❌ 主動式漏洞掃描（Active Scanning）在 OT 環境有風險，可能造成設備異常

## 補充主題八：OWASP LLM Top 10

| 排名 | 弱點 | 說明 |
|---|---|---|
| LLM01 | 提示詞注入（Prompt Injection） | 直接或間接操控 LLM 行為 |
| LLM02 | 不安全的輸出處理 | LLM 輸出未驗證直接使用 |
| LLM03 | 訓練資料中毒 | 污染訓練資料影響模型行為 |
| LLM06 | 敏感資訊揭露 | LLM 洩露訓練資料或使用者輸入 |
| LLM08 | 過度代理 | LLM 被授予過多工具或權限 |

**間接提示詞注入（Indirect Prompt Injection）**：攻擊者將惡意指令嵌入 LLM 會讀取的外部資料（如網頁、文件），包含供應鏈外部來源。

**防禦**：
- 對高風險操作加入人工審核（Human-in-the-loop）
- 最小權限原則（不提供不必要的工具）
- 定期滲透測試紅隊演練，將 LLM 視為不受信任實體
- 企業 AI：使用地端部署或私有雲 LLM + 資料不落地政策

## 補充主題九：AI 威脅與 Deepfake 防禦

### Deepfake 商業詐騙（BEC + Deepfake）
- 攻擊者使用 AI Deepfake 偽造高階主管視訊/語音，要求緊急匯款。
- **最後一道防線（流程面）**：**外部驗證（Out-of-Band Verification）**，透過第二個不同管道（如公司內部加密通訊）確認指示。

### 防禦策略
- 建立雙人簽核（Dual Control）財務流程
- 定期社交工程演練，納入 Deepfake/AI 語音情境
- 採購 Deepfake 偵測解決方案（分析眨眼頻率、語音合成痕跡）
- **風險評估調整**：GenAI 普及降低攻擊成本，應**調升** Deepfake 攻擊的機率評估

## 補充主題十：EDR / NDR / XDR / MDR 比較

| 工具 | 全名 | 範疇 | 特點 |
|---|---|---|---|
| **EDR** | Endpoint Detection & Response | 端點 | 行為分析、即時隔離、記憶體注入偵測 |
| **NDR** | Network Detection & Response | 網路 | 流量分析、橫向移動偵測 |
| **XDR** | Extended Detection & Response | 跨來源 | 整合 EDR+NDR+身分，關聯分析 |
| **MDR** | Managed Detection & Response | 委外服務 | 24×7 監控、即時回應，補足夜間/假日人力 |

**情境應用**：
- 端點行為異常 → EDR 主要處置
- 跨端點、網路、身分事件關聯 → XDR
- 夜間無人值守 → MDR（外部服務）提供即時回應
- 網路流量未有異常但端點異常 → NDR 效益有限

## 補充主題十一：ISO 27001:2022 威脅情資（Threat Intelligence）

ISO 27001:2022 新增控制措施 **5.7 威脅情資（Threat Intelligence）**：

實務作業流程：
1. 訂閱並分析來自 **ISAC（資安資訊分享與分析中心）** 或廠商的情資
2. 將情資與內部環境進行關聯分析，判斷相關性與衝擊
3. 依據情資調整防禦策略（如更新 WAF 規則、封鎖惡意 IP）
4. ❌ 不能只收集內部日誌，需參考外部情資

**公務機關 IT 資安治理成熟度評估（數位部資安署）**：共 **6 個成熟度類級**（0~5 或依指引定義）。
