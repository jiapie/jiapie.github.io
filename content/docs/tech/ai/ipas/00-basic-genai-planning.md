## 1. No Code / Low Code 概念

### 定義與比較

| | **No Code** | **Low Code** |
|--|------------|-------------|
| **目標使用者** | 非技術背景人員 | 具備技術背景的開發者 |
| **操作方式** | 視覺化介面、拖放操作，零程式碼 | 視覺化工具＋少量程式碼擴充 |
| **適用場景** | 快速原型、小型應用、內部工具 | 中大型企業、複雜邏輯應用 |
| **代表工具** | make.com、Bubble、Glide、Power Apps | Retool、Copilot Studio、GitHub Copilot |
| **優勢** | 降低技術門檻、加速開發 | 客製化彈性、跨平台整合 |
| **限制** | 功能彈性受限、不適合高安全性任務 | 仍需一定技術能力 |

### 生成式 AI 結合 No Code / Low Code

生成式 AI（如 ChatGPT、DALL-E）進一步提升平台效能：
- **自動生成程式碼**：協助開發客製化功能
- **模板設計優化**：AI 提供介面設計與流程建議
- **數據分析與決策**：自動化分析並提供建議
- **行銷文案生成**：輸入產品資訊即時產出行銷內容

### 選擇平台的六大考量

1. **目標用戶與技術需求**：No Code vs Low Code 的適用對象
2. **功能與擴展性**：系統整合（CRM、ERP）、高度自訂性
3. **安全性與合規性**：資料加密、身份驗證、符合個資法
4. **成本與效益**：TCO（總擁有成本）+ ROI（投資回報率）
5. **技術支援與社群資源**：文件、論壇、客服支援品質
6. **市場評價與成功案例**：相似產業的導入案例

## 3. 提示工程（Prompt Engineering）

**核心技巧：**

| 技巧 | 說明 | 範例 |
|------|------|------|
| **明確性與上下文** | 清楚定義任務與角色，提供背景資訊 | 「扮演金融分析師，解釋 ROC 曲線在風險評估的應用」 |
| **Few-shot Prompting** | 提供 1–2 個範例供 AI 模仿 | 附上輸入/輸出格式範例 |
| **Chain-of-Thought（CoT）** | 加入「請一步一步思考」，提升邏輯推理準確度 | 適合數學、邏輯推理題 |
| **結構化格式** | 指定輸出格式（表格、條列、JSON、字數限制）| 「請用繁體中文、條列式，控制在 200 字內」 |
| **迭代優化** | 根據輸出結果持續修改關鍵字與結構 | A/B 測試不同 Prompt 的效果 |

**文生圖 Prompt 優化：**
- **加入更多細節**：材質、光影、背景、動作，縮小 AI 猜測空間
- **指定風格**：賽博朋克、寫實攝影、吉卜力動漫風
- **嘗試不同描述方式**：換同義詞、調整句型結構（長句拆成標籤）

## 4. 生成式 AI 導入評估規劃

### 導入五大階段

```
1. 需求分析 → 2. 設計與規劃 → 3. 驗證 POC → 4. 實施部署 → 5. 營運監控
```

| 階段 | 核心工作 |
|------|---------|
| **需求分析** | 釐清業務痛點、定義 KPI、資料盤點、確認法規合規 |
| **設計與規劃** | 技術選型（模型/平台/工具）、資料流架構、地端 vs 雲端 |
| **驗證 POC** | 小規模試驗驗證技術可行性；若不符需求快速調整 |
| **實施部署** | 模型部署（地端/雲端/邊緣）、自動化數據流程建置 |
| **營運監控** | 效能監控、資料飄移偵測、持續優化、倫理審查 |

### AI 導入可行性評估三面向

**業務需求分析：**
- 哪些流程重複性高、規則清晰、數據驅動？
- 是否有流程瓶頸或服務缺口？
- 要提升哪些 KPI（效率、準確率、回應速度）？

**技術可行性：**
- 資料品質是否足夠（完整性、一致性、標注品質）？
- IT 基礎設施（伺服器、雲端、資料倉儲）是否就緒？
- 適合何種學習方式（監督/非監督/生成式）？

**成本效益評估：**

| 財務指標 | 計算公式 | 說明 |
|---------|---------|------|
| **ROI**（投資回報率） | (回報 - 成本) / 成本 × 100% | 越高代表每投入資金回報越多 |
| **NPV**（淨現值） | Σ(現金流 / (1+折現率)^n) - 初期投資 | NPV > 0 表示專案有正向價值 |
| **回收期** | 初期投資成本 / 每年淨現金流 | 越短代表風險越低 |

### 生成式 AI 風險管理

**主要風險類型：**

| 風險類型 | 說明 | 因應措施 |
|---------|------|---------|
| **模型準確性** | AI 可能產生幻覺（Hallucination）或不準確內容 | 多層次驗證、人工審核、RAG |
| **資料隱私安全** | 敏感資料洩漏、違反個資法 | 加密、匿名化、權限控管 |
| **道德與倫理** | 生成偏見內容、假消息 | 偏見偵測與修正機制 |
| **智慧財產權** | 訓練資料來源與生成內容版權歸屬 | 確認授權、內容審核機制 |
| **系統安全** | 對抗性攻擊、資料外洩 | 零信任架構（ZTA）、監控告警 |

**AI 導入不當的常見問題：**
- 缺乏數位/資料成熟度 → 先建立 ETL 與資料治理
- 跨部門協作不足 → 設立 AI 專案辦公室、指定負責人
- 預算規劃不完整 → 評估 TCO，規劃試辦環境

## 5. RAG 與 Fine-tuning 比較

| | **RAG（檢索增強生成）** | **Fine-tuning（微調）** |
|--|----------------------|----------------------|
| **原理** | 不改變模型參數，從外部知識庫檢索資料後生成 | 改變模型內部權重，讓模型學會特定知識/風格 |
| **成本** | 低（無需重新訓練） | 高（需高品質標注資料＋算力）|
| **即時性** | 高（知識庫即時更新）| 低（需重新訓練才能更新知識）|
| **適合場景** | 知識更新快的領域（法律、金融、客服）| 固定專業領域任務（醫療診斷、專業翻譯）|
| **風險** | 檢索錯誤可能影響生成品質 | 過擬合、遺災性遺忘（Catastrophic Forgetting）|

**最佳實務：** 大多數產業應用中，RAG 與 Fine-tuning 混合使用，兼顧即時性與專業性。
